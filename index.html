🧠 IntelliFix — AI-Powered VS Code Extension

Organization: School of Science, Engineering & Technology — SSET

Duration: March 2025 – October 2025 (8 months)

Team Size: 5 members | Role: Product Lead & Developer

Type: Internship Project

Product Strategy User Research Agile/Scrum Launch Published on VS Code Marketplace

Discover: Problem Identification & Research

Conducted comprehensive user research with 20+ beginner developers through 1-on-1 interviews and surveys (50+ responses) across 3 universities. Applied structured research methodologies including observational studies and pain point analysis to identify critical blockers in developer learning journey.

Key Findings:

  • 73% of users spent 30+ minutes on single error messages
  • 60% considered quitting programming due to cryptic error messages
  • Average debugging time: 45 minutes per error for beginners
  • Primary frustration: Lack of actionable solutions and educational context
Key Insight: "I wish error messages could just explain what's wrong in simple terms and show me how to fix it" — CS Student

Competitive Analysis: Evaluated Error Lens (inline display only), GitHub Copilot (no educational focus), and VS Code IntelliSense (no error learning). Identified market gap: no tool combined AI explanations with educational scaffolding for beginners.

Define: Product Strategy & Planning

Product Vision: Transform error messages from frustration points into learning opportunities, empowering beginner developers to learn faster with confidence.

Target User Persona:

  • Primary: CS students & bootcamp learners (0-2 years experience) struggling with cryptic errors
  • Secondary: Coding instructors needing tools to help students debug independently

Feature Prioritization (RICE Framework):

  1. One-Click Fix (RICE: 600) — High reach, moderate impact, low effort
  2. AI Error Explanations (RICE: 400) — High reach, high impact, moderate effort
  3. Learn from Error Module (RICE: 131) — Medium reach, high impact on retention, high effort

Success Metrics Defined:

  • Primary: 60% reduction in time to resolve errors
  • Activation rate: 40% within first week
  • Feature usage: 10+ uses per week per active user
  • User satisfaction: NPS score tracking

Build: Agile Execution & Cross-Functional Leadership

Agile Methodology: Led 8 two-week sprints with structured planning, daily standups, and retrospectives.

Sprint Highlights:

  • Sprint 1-2: Foundation & error detection engine (4 languages support)
  • Sprint 3-4: AI integration — explanation engine with <3 sec response time, 90%+ accuracy
  • Sprint 5-6: Interactive features — one-click fix with 80%+ accuracy, undo capability
  • Sprint 7-8: Launch prep — documentation, onboarding flow, beta testing

Cross-Functional Team Management:

  • Coordinated 2 Frontend + 2 Backend engineers, aligned technical execution with product vision
  • Wrote 40+ user stories with clear acceptance criteria
  • Managed prioritized backlog of 50+ items across 8 sprints
  • Facilitated weekly sprint reviews with faculty advisor and monthly stakeholder presentations
  • Resolved scope conflicts through data-driven prioritization and transparent communication

Stakeholder Management: Balanced needs of users (simplicity), engineers (technical feasibility), and academic stakeholders (learning outcomes) through clear PRDs and regular alignment meetings.

Launch: Go-to-Market Strategy & Beta Testing

Beta Testing Program:

  • Recruited 50+ early adopters from 3 universities
  • Conducted 3 rounds of usability testing
  • Collected 200+ pieces of feedback through surveys and interviews
  • Key iteration: Simplified onboarding from 5 to 3 steps (50% dropout reduction)

Launch Strategy:

  • Published to VS Code Marketplace (10M+ monthly active users)
  • Created demo video, Medium article, and comprehensive documentation
  • Distributed through developer communities (Reddit, Discord) and coding bootcamps
  • Established feedback channels (GitHub Issues, user surveys) for continuous improvement

Documentation Strategy: Invested 20% of dev time creating README, video tutorials, FAQ, and troubleshooting guides — resulting in 40% activation rate (15% increase from initial 25%).

Iterate: Post-Launch Optimization

3 Major Iterations Based on User Data:

Iteration 1 — Learn Module Redesign:

  • Problem: Only 30% engagement with Learn Module
  • Analysis: User feedback revealed interface complexity and lack of progress tracking
  • Solution: Simplified UI, added progress indicators and gamification elements
  • Result: Engagement increased to 45% (+50% improvement)

Iteration 2 — Performance Optimization:

  • Problem: 3-5 second delay for AI explanations (user complaints)
  • Solution: Implemented caching and optimized API calls
  • Result: Response time reduced to <2 seconds (60% improvement)

Iteration 3 — Language Support Expansion:

  • Problem: User requests for additional language support
  • Solution: Added C++ and Java support based on demand analysis
  • Result: User base expanded by 25%

Learn: Key Takeaways

  • User-centric approach wins: Features built from validated user pain points had 3x higher adoption than assumed features
  • Data-driven decisions prevent waste: Weekly metric reviews and A/B testing prevented building wrong features and guided optimization efforts
  • Iteration beats perfection: Shipping MVP and iterating based on real usage data was more effective than extended planning
  • Documentation is product: Clear documentation directly correlated with 15% increase in activation rate and 2x higher retention
  • Cross-functional alignment is critical: Regular communication and clear PRDs prevented misalignment and ensured on-time delivery across 8 sprints
  • Technical understanding enables better PM: CS background helped evaluate feasibility, make realistic commitments, and earn engineering team trust

🏫 University Equipment Lending Management System

Type: Academic Project

Role: System Analyst & Developer

System Analysis Requirements Engineering Architecture Design Object-Oriented Design

Discover: System Requirements

Analyzed requirements for managing university equipment inventory and lending workflows. Identified key challenges: equipment tracking across multiple departments, complex permission systems, and manual reservation processes causing delays and lost equipment.

Key Challenge: Design a system that balances ease-of-use for students with robust tracking for administrators

Build: System Design & Architecture

  • User Role Definition: Defined three primary user roles with distinct permissions:
    • Students: Browse equipment, make reservations, track lending history
    • Staff: Manage equipment inventory, approve/reject requests, track returns
    • Administrators: System configuration, user management, analytics and reporting
  • Object-Oriented Architecture: Designed system using OOP principles to ensure scalability and maintainability
  • System Workflows: Documented complete workflows for equipment reservation, lending, return, and maintenance processes
  • Data Modeling: Created comprehensive data models showing relationships between users, equipment, reservations, and transactions

Launch: Key Features Implemented

Equipment Catalog

Searchable inventory with availability status and specifications

Reservation System

Online booking with conflict prevention and automatic notifications

Permission Management

Role-based access control for different user types

Lending Tracking

Complete audit trail of who borrowed what and when

Reporting Dashboard

Analytics on equipment utilization and lending patterns

Maintenance Scheduling

Track equipment condition and schedule preventive maintenance

Measure: Technical Decisions

  • Requirements Documentation: Created detailed functional and non-functional requirements for development team
  • Scalability Planning: Designed architecture to handle growth from single department to university-wide deployment
  • Testing & Validation: Implemented and tested core features to ensure alignment with requirements

Learn: Key Takeaways

  • Requirements clarity is critical: Detailed upfront analysis prevented scope creep and rework
  • Multi-user systems require careful permission design: Balancing access for different roles was more complex than anticipated
  • Documentation enables collaboration: Clear specifications allowed smooth handoff between analysis and implementation phases

🏠 RENTMATE — Full-Stack Property Rental Platform

Type: Academic Project (Solo)

Role: Full-Stack Developer & Product Designer

Product Design User Research Technical Implementation Full-Stack

Discover: Market Opportunity

Identified inefficiencies in the property rental market for students and landlords. Key problems: lack of transparency in pricing, difficult communication between parties, and manual document handling.

Opportunity: Build a two-sided marketplace that streamlines the entire rental process

Build: Product Development

  • User Personas: Defined two primary users — Students (renters) and Landlords (property owners)
  • Feature Mapping: Created user journey maps for both personas to identify must-have features:
    • For Students: Property search with filters, document upload, payment tracking
    • For Landlords: Property listing, tenant management, rent request handling
  • Technical Architecture: Designed and implemented scalable architecture independently:
    • Backend: Spring Boot + PostgreSQL for data persistence
    • Frontend: React for dynamic UI
    • Security: Spring Security with BCrypt for authentication
    • API Documentation: Swagger for clear API specifications
  • Solo Development: Managed entire development lifecycle from ideation to deployment, handling both product decisions and technical implementation

Launch: Key Features Delivered

Authentication System

Secure registration & login for users and landlords

Property Management

Full CRUD operations with image upload support

Search & Filter

Advanced filtering by location, price, amenities

Document Handling

Digital document upload and management system

Payment Tracking

Track rent payments and request history

Messaging System

Real-time chat between renters and landlords

Measure: Technical & Product Decisions

  • Database Design: Normalized schema to ensure data integrity and scalability
  • API-First Approach: RESTful API design enabled future mobile app development
  • User Testing: Conducted usability testing with 10 users, leading to UI refinements
  • Performance: Optimized queries and implemented caching for faster load times

Learn: Product Insights

  • Two-sided marketplaces are complex: Balancing features for both user types required careful prioritization — a valuable lesson in making product trade-offs independently
  • Start simple, iterate: Launched with core features first, planned advanced features (reviews, analytics) for v2.0
  • Technical trade-offs: Learned to balance ideal solutions with what's practical given time/resources when working solo
  • End-to-end ownership: Managing both product and technical aspects deepened my understanding of the full development lifecycle
  • User feedback is gold: Early testing revealed critical UX issues that wouldn't have been obvious during solo development
